## Summary

This feature implements a pluggable adapter interface (`IDocumentSource`) for reading document content and subscribing to directory changes from multiple sources (file system, Google Drive). The design uses `IAsyncEnumerable<T>` streaming patterns for efficient, async-first operations with cancellation support. Two concrete adapters will be implemented: file system watchers for local directories and Google Drive API integration for cloud-based content synchronization.

## Technical Context

**Language/Version**: C# 11 with .NET 8.0  
**Primary Dependencies**: 
  - Google.Apis.Drive.v3 (for Google Drive adapter)
  - System.IO.FileSystemWatcher (for file system monitoring)
  - xUnit + Moq (for testing)
**Storage**: N/A (this is a content source adapter layer, not a storage provider)  
**Testing**: xUnit with Moq for unit tests; integration tests using temporary directories and mock Google Drive responses  
**Target Platform**: Windows/macOS/Linux (cross-platform .NET 8)  
**Project Type**: Library (adapter layer with plugin architecture)  
**Performance Goals**: 
  - ReadAllAsync: Stream 100+ documents in < 2 seconds
  - SubscribeToChangesAsync: Detect and yield changes within 1 second
  - Support 100+ concurrent streams without degradation
**Constraints**: 
  - < 2 seconds latency for file system change detection
  - < 3 seconds authentication and first stream for Google Drive
  - Graceful error recovery for network interruptions
  - Support cancellation via CancellationToken
**Scale/Scope**: 
  - Initial: Support 1-2 sources per session
  - Future: Multiple concurrent sources with independent subscriptions

## Phase 0: Research

**Status**: Complete  
**Output**: See [research.md](research.md)

### Known Clarifications (from Spec Session 2026-05-02)

All architectural questions have been addressed through the specification clarification process. The decisions below are documented in the spec and inform the Phase 1 design:

1. **Configuration Design**: `DocumentSourceConfig` with base properties + adapter-specific dictionary
2. **Async Pattern**: `IAsyncEnumerable<T>` with `CancellationToken` for modern C# async patterns
3. **Streaming vs. Batch**: Streaming with `IAsyncEnumerable` for memory efficiency
4. **Adapter Extensibility**: Generic config with dictionary to avoid tight coupling
5. **Change Notifications**: Include full `DocumentContent` for immediate processing

### Research Findings

**File System Implementation**:
- Use `System.IO.FileSystemWatcher` for real-time change detection (available on Windows/macOS/Linux)
- Challenge: FileSystemWatcher can miss rapid changes; mitigate with debouncing (100-200ms)
- Advantage: No external dependencies, built-in to .NET framework
- Testing: Use temporary directories with controlled file operations

**Google Drive Implementation**:
- Google Workspace API provides two approaches:
  - Option 1: Google Drive API v3 with polling (simpler, sufficient for MVP)
  - Option 2: Google Watch with webhooks (complex, requires public endpoint)
- **Decision**: Use polling via Google.Apis.Drive.v3 NuGet package for MVP
- Authentication: OAuth2 via Google.Apis.Auth; assume token already obtained by host application
- Limitation: Polling introduces 5-10 second change latency (vs. <1s for file system)
- Testing: Use mock Google Drive responses; no actual API calls in unit tests

**Error Handling Strategy**:
- File system: File access exceptions, permission denied, path not found
- Google Drive: Network timeout, auth failure, quota exceeded, folder deleted
- All errors should be yielded as `DocumentChange` with error flag OR thrown with descriptive messages
- Decision: Throw exceptions for recoverable errors (let consumer decide to retry); yield error notifications for transient issues

**Performance Considerations**:
- FileSystemWatcher can monitor large directories efficiently
- Google Drive polling: Start with 5-second intervals; adjust based on usage patterns
- `IAsyncEnumerable` with `yield return` ensures low memory footprint for large document sets
- CancellationToken support critical for cleanup (dispose watchers, cancel API requests)

### Phase 1a: Data Model

See [data-model.md](data-model.md) for complete entity definitions. Summary:

**Core Entities**:

1. **IDocumentSource** (interface)
   - Method: `ReadAllAsync(CancellationToken)` → `IAsyncEnumerable<DocumentContent>`
   - Method: `SubscribeToChangesAsync(CancellationToken)` → `IAsyncEnumerable<DocumentChange>`

2. **DocumentSourceConfig** (class)
   - Properties: `AdapterType` (enum: FileSystem|GoogleDrive), `SourceId`, `Configuration` (Dictionary<string, object>)
   - Adapter-specific keys: FileSystem → `DirectoryPath`, `Extension`; GoogleDrive → `FolderId`, `DriveType`

3. **DocumentContent** (record)
   - Properties: `Id` (string), `Content` (string), `LastModified` (DateTime), `Encoding` (string)

4. **DocumentChange** (record)
   - Properties: `ChangeType` (enum: Created|Modified|Deleted), `Document` (DocumentContent), `Timestamp` (DateTime)

5. **DocumentChangeType** (enum)
   - Values: Created, Modified, Deleted

### Phase 1b: Interface Contracts

See [contracts/](contracts/) for detailed specifications:

- **idocument-source-contract.md**: Full method signatures, parameters, return types, exceptions
- **document-content-contract.md**: Field definitions, validation rules, encoding specifications
- **document-change-contract.md**: Change event structure, payload requirements

### Phase 1c: Architecture Design

**Adapter Pattern**:
- `IDocumentSource` interface lives in `Application/Ports/`
- Each adapter implements `IDocumentSource` and handles initialization via `IDocumentSourceFactory` or direct instantiation
- Adapters expose no dependencies to consumers; all Google Drive/FileSystemWatcher logic is encapsulated

**Error Handling Model**:
- Thrown exceptions: Authentication failures, configuration errors, access denied
- Streamed errors: Network timeouts (will be retried), transient API failures (yield as change notifications)

**Testing Strategy**:
- Unit tests: Mock dependencies (Google Drive API, file system operations)
- Integration tests: Real temporary directories, mock Google Drive responses
- Contract tests: Verify both adapters fulfill `IDocumentSource` contract
